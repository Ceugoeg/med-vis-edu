# Phase 2 核心蓝图：边缘感知迁移与多语言生态构建

## 1. 阶段最终目标
本阶段（Phase 2）的核心目标是彻底落实“Cloud-Edge-End”解耦架构，将视觉感知与推理计算完全下放至边缘端设备（Pico Max 与 SC3336 摄像头），释放前端渲染设备的算力。同时，丰富系统的医学展示内容，实现全链路的国际化（i18n）支持。最终交付物需满足架构设计的严格性能指标（子 60ms 延迟，60 FPS 推流），并具备展示多种解剖结构与多语言查询的完整演示能力。

## 2. 成员任务拆解与交接链路

### 成员 C：边缘端感知 MVP 开发与基准测试（前置探索）
作为边缘端迁移的先头部队，首要任务是在 Pico Max 上构建纯 Python 版本的最小可行性产品（MVP），跑通从硬件采集到网络下发的全流程，并收集消融实验所需的对比数据。
首先需打通 Pico Max 与微雪 SC3336 CSI 摄像头的视频流抓取机制。随后，在 Python 环境中运行 MediaPipe Hands，并严格复刻 Phase 1 中 JavaScript 版本的自适应指数移动平均滤波（Adaptive EMA）与层次化状态机（HSM）启发式手势判定逻辑。在数据通信方面，需建立轻量级网络服务（如 WebSocket），将结构化的 JSON 数据（包含离散手势状态、21 个 3D 归一化坐标及运动置信度）下发到前端渲染环境。最后，在此未经深度优化的纯 CPU 运行环境下，详细记录帧率（FPS）、端到端延迟与 CPU 占用率，作为证明后续 NPU 加速必要性的 Benchmark 数据，并将验证完毕的逻辑与数据打包移交至成员 A。

### 成员 A：底层重写与 NPU 异构加速（性能攻坚）
在接收成员 C 跑通的 Python MVP 及通信数据字典后，核心任务是利用底层语言与硬件特性打破运算瓶颈，实现工程落地。
需全面摒弃 Python 层的性能损耗，使用 C++ 结合 MediaPipe 原生 C++ API 重构整个视觉采集、滤波与状态判定管线。进一步的技术挑战在于异构计算，需将 MediaPipe 底层的 TFLite 推理模型转化为瑞芯微专用的 RKNN 格式，直接调用 RV1106 主控内建的 0.5 TOPS NPU 进行张量加速。最终需保证边缘节点能够长期稳定地输出 60 FPS 的结构化感知数据，确保整体交互回路的端到端延迟被严格压缩在 60ms 以内，彻底消除空间操作的迟滞感。

### 成员 B：3D 资产扩充与系统级多语言适配（内容生态）
专注于提升系统的医学教育展示丰度与国际化交互体验，实现从模型底层资产到应用层 UI 的全维多语言支持。
在三维资产构建上，需制作、导出并整合更多高精度的医学解剖模型，且必须严格遵守引擎层约定的 Mesh 节点命名规范以防碰撞失效。在国际化数据结构设计上，需建立标准化的 JSON 本地化配置词典（如继续扩充 `assets/locales/` 下的结构化文件），将医学实体的 ID 映射至对应的多语言术语、介绍文本及供大模型检索的 Query 提示词。最后，需全面重构前端页面的多语言逻辑，打通 UI 框架与渲染引擎的文字展示，确保侧边栏、交互提示词与 3D 资产的语言状态保持严格同步。系统启动时的默认语言需强制设定为英文（English），并提供全局无缝热切换能力。
